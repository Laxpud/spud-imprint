# 开发规范

本文档记录 Laxpud Imprint 仓库的长期协作约定。除非有明确原因，后续代码、文档和测试都应遵守这里的规则。

## README 语言

- `README.md` 是默认入口，使用中文。
- `README.en.md` 是英文版本。
- 当 README 内容发生实质变化时，中英文版本都要同步更新。
- 面向 GitHub 首页的介绍优先写在中文 README 中，英文 README 保持同等信息量，但不要求逐字翻译。

## Python 环境管理

推荐使用 `uv` 管理本项目的 Python 虚拟环境和依赖：

```powershell
uv venv
uv pip install -e .
```

推荐 `uv` 的原因：

- 创建虚拟环境和安装依赖速度快。
- 命令清晰，适合后续接入 CI 和自动化脚本。
- 不改变项目本身的 Python 包结构，仍然兼容普通 `pip` 和 `venv`。

`uv` 是推荐工具，不是运行时依赖。最终用户使用打包好的 CLI 或 GUI 时，不应该需要安装 `uv`。

## 代码注释

- Python 注释和 docstring 使用中文。
- 注释应符合 PEP 8 / PEP 257 的基本风格：简洁、准确、解释意图。
- 面向初学者不等于给每一行写重复注释。优先解释数据流、坐标计算、图像模式转换、配置含义和非显然的边界处理。
- 公共函数、公共类、复杂私有方法应有 docstring。
- 简单赋值、显而易见的调用不需要注释。

示例：

```python
def mm_to_px(self, mm):
    """将毫米转换为像素，换算依据是原始照片的 DPI。"""
    return int(mm * self.original_dpi / 25.4)
```

## 命名和输出语言

- 代码标识符使用英文，例如变量名、函数名、类名、模块名。
- 配置键名使用英文，例如 `input_dir`、`margin_relative`。
- CLI 命令、日志、异常消息、终端输出使用英文。
- Python 注释和 docstring 使用中文。
- 用户文档可以按目标读者选择中文或英文；默认文档优先中文。

这个规则的目的，是让代码和工具接口保持国际化和工程通用性，同时让中文读者更容易理解实现逻辑。

## 测试素材

当前测试主要使用程序生成的小图片和临时目录，不依赖真实照片素材。

真实照片素材不应该默认作为测试的一部分直接进入 Git，原因包括：

- 文件体积大，会让仓库和克隆速度变慢。
- EXIF 可能包含设备、时间、位置等隐私信息。
- 照片可能有版权或授权问题。
- 大图测试会拖慢普通单元测试。

允许提交的测试素材应满足这些条件：

- 体积小，通常建议单个文件小于 200 KB。
- 已脱敏，不包含个人隐私、位置或敏感 EXIF。
- 来源清楚，可自由分发。
- 用途明确，放在 `tests/fixtures/` 下。

如果需要测试真实 EXIF 行为，优先创建小型 fixture，而不是直接使用个人原片。可以单独准备一张小尺寸、脱敏但保留必要 EXIF 字段的 JPEG。

## 测试分层

- 单元测试可以使用程序生成的图片，验证纯逻辑和小范围图像处理。
- fixture 测试用于验证固定 EXIF、字体、特殊图片模式等稳定样例。
- 大图、性能、真实照片批处理应作为手动测试或单独的慢测试，不默认进入快速测试集。

默认测试命令：

```powershell
python -m unittest discover -s tests
```

## Git 管理

- 版本演进交给 Git commit 和 tag，不再创建 `v0.5.py` 这类版本文件。
- 旧实现保留在 `legacy/` 中作为迁移参考。
- 原始工作照片、生成输出、临时目录不提交。
- 提交前至少运行默认测试。
